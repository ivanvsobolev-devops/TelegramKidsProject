# ADR-010: Flutter for MVP Mobile Apps

## Status

Proposed

Superseded by ADR-013 for the bot-first MVP.

## Context

The repository bootstrap originally required a Flutter child app and a Flutter parent app. Earlier architecture work preferred native Android for MVP because TDLib and Android platform integration are important for the child client. The current implementation direction prioritizes a Flutter repository scaffold for the child app only, while the parent app is deferred to V2.

Flutter can still support Android-first delivery while preserving a path to iOS later. TDLib and platform-specific integration will require native Android platform channels or Flutter plugins when business logic is added.

## Decision

Use Flutter for the MVP child application scaffold.

The child app will live under `apps/child_app`. The scaffold contains no business logic. Platform-specific Android files are generated by Flutter tooling during bootstrap/build rather than hand-authored in the initial repository bootstrap.

## Alternatives Considered

### Flutter Child App Only

Pros:

- One UI framework for the child app.
- Preserves future iOS path.
- Fast app shell and UI iteration.
- Matches the repository bootstrap requirement.

Cons:

- TDLib integration will require native platform integration.
- Deep links, background execution, notifications, and secure storage still need Android-specific work.
- Child app safety-critical paths must be carefully isolated from UI code.

### Native Android

Pros:

- Direct Android platform and TDLib integration.
- Lower platform-channel complexity for the child app.
- Best fit for Android-specific background, notification, and secure-storage behavior.

Cons:

- Does not match the requested Flutter bootstrap.
- Less code reuse if iOS becomes a near-term target.
- Requires separate mobile architecture if parent app moves cross-platform later.

### Native Child App and Flutter Parent App

Pros:

- Keeps TDLib-heavy child app native.
- Allows parent approval UI to be cross-platform in a later version.

Cons:

- Introduces two mobile stacks.
- Increases CI, release, and QA complexity.
- Not aligned with a simple repository bootstrap.

## Consequences

- Flutter is the mobile scaffold standard for the MVP child app.
- Future Telegram integration work must define native platform-channel boundaries.
- The child app remains untrusted; backend approval state remains authoritative.
- Android platform files may be generated by Flutter tooling during bootstrap/build until the project needs committed native customizations.

## Implementation Notes

- Keep Flutter app shells free of business logic in the bootstrap.
- Add Android platform code only when needed for TDLib, notifications, deep links, or secure storage.
- Document native-channel APIs before implementing Telegram integration.
- Revisit committed platform directories once the apps require native customizations.
